# Incident Communication

Effective communication is essential during incident management. Clear, timely, and accurate updates help stakeholders understand the situation, reduce confusion, and ensure coordinated efforts to restore services.

The goal is to keep everyone informed while the technical team focuses on resolving the incident.

---

## Objectives

- Inform stakeholders about the incident
- Provide regular progress updates
- Set realistic expectations
- Coordinate between support teams
- Maintain transparency until resolution

---

## Stakeholders

Depending on the incident, communication may involve:

- End Users
- Service Desk
- Application Support Team
- Infrastructure Team
- Database Team
- Network Team
- Business Owners
- Incident Manager
- Management

---

## Communication Lifecycle

```text
Incident Detected
        │
        ▼
Initial Notification
        │
        ▼
Investigation Update
        │
        ▼
Regular Status Updates
        │
        ▼
Resolution Notification
        │
        ▼
Incident Closure
```

---

## Initial Notification

The first communication should be sent as soon as the incident is confirmed.

Include:

- Incident ID
- Affected application or service
- Time detected
- Brief description
- Business impact
- Current status
- Assigned support team

### Example

```
Incident ID : INC00012345

Application : Online Banking

Issue:
Users are unable to log in to the application.

Impact:
All customers are affected.

Status:
Investigation is in progress.

Next Update:
Within 30 minutes.
```

---

## Progress Updates

Provide updates at regular intervals, even if there is no major progress.

Include:

- Current status
- Actions completed
- Findings
- Remaining work
- Expected next update

### Example

```
Current Status:
Application team has identified a database connectivity issue.

Action Taken:
Database team is investigating the connection failure.

ETA:
15 minutes.

Next Update:
12:30 PM
```

---

## Resolution Communication

After the issue is resolved, communicate:

- Resolution time
- Cause of the issue
- Services restored
- Any user action required

### Example

```
Status:
Resolved

Resolution:
The database service was restarted successfully.

Impact:
Application access has been restored.

Users may now log in normally.
```

---

## Closure Communication

When the incident is officially closed, summarize:

- Incident timeline
- Root cause (if available)
- Resolution
- Preventive actions
- Ticket closure confirmation

## Major Incident Communication

For critical incidents:

- Start a bridge call immediately.
- Notify senior management.
- Provide updates at defined intervals (for example, every 15–30 minutes).
- Record key decisions and action items.
- Ensure all technical teams remain aligned.

---

## Communication Best Practices

- Use simple and professional language.
- Share only verified information.
- Avoid speculation about the root cause.
- Provide regular updates, even if there is no change.
- Clearly mention the next expected update.
- Keep all communications concise and factual.
- Document important decisions in the incident ticket.

---

## Common Mistakes

- Delaying the first notification
- Sharing unverified information
- Missing scheduled updates
- Using technical jargon with business users
- Closing communication before user confirmation
- Failing to document important updates

## Key Takeaways

- Communication is as important as technical troubleshooting.
- Keep stakeholders informed throughout the incident lifecycle.
- Share accurate, timely, and concise updates.
- Regular communication builds trust and improves incident management.
